# QuickShell desktop shell requirements

## Observations

- [fact] Desktop shell in `configs/quickshell/` covering common day-to-day actions
- [decision] Shell should feel compact, flat, dark, square, and information-dense
- [technique] Reuse small set of shared QML primitives instead of implementing each dashboard tab independently
- [requirement] Missing hardware must degrade gracefully; no backend must break the shell

## Goal

Build a desktop shell in `configs/quickshell/` that covers the most common day-to-day actions out of the box while staying visually close to the current shell.

The shell should feel compact, flat, dark, square, and information-dense. It should reuse a small set of shared QML primitives instead of implementing each dashboard tab independently.

## Current style anchors

- Keep the current bar feel: dark background, thin accent separator, compact height, and no decorative chrome.
- Keep the current flat design rules from [[quickshell-design]].
- Keep square panels by default. Rounded corners remain opt-in through `Config.styling.rounded`.
- Keep subtle interaction motion only for feedback, not decoration.

## General rules

- Use the Catppuccin palette generated by `modules/stylix/quickshell.nix` and loaded at runtime from `quickshell/catppuccin-palette.json`.
- Read colors through `Config.colors`, `Config.styling`, and `Config.palette`.
- Do not introduce colors outside the Catppuccin palette except through opacity adjustments derived from existing colors.
- Follow flat design principles.
- Do not use gradients, drop shadows, glass, or 3D effects.
- Use the spacing scale from `Config.spacing`.
- Use `Config.spacing.xxs` and `Config.spacing.xs` for compact bar and row internals.
- Use `Config.spacing.sm` and `Config.spacing.md` for dashboard sections.
- Keep text, icons, sliders, rows, and badges aligned to the 4px base grid.
- Keep animations subtle and fast.
- Use `100-250ms` for micro-interactions and up to `400ms` for sidebar open or close transitions.
- Respect `Config.behaviour.animation.enabled` and `Config.behaviour.animation.calc(...)`.
- Prioritize clarity, fast scanability, and predictable interactions over novelty.
- Missing hardware must degrade gracefully. No battery, bluetooth, GPU, or brightness backend must not break the shell.

## Interaction goals

- Most common actions should take at most two clicks from the bar.
- The dashboard opens on the focused screen.
- `Esc` closes the dashboard unless an input, popup, or drag interaction owns focus.
- Clicking outside the dashboard closes it unless a modal or drag interaction is active.
- Switching tabs must not close the dashboard.
- Expanded rows should not reorder surrounding content unexpectedly.
- Destructive actions require confirmation.

## Shared state

Create one shared state machine for both the bar and the dashboard.

Required state:

- `dashboardOpen: bool`
- `activeTab: "overview" | "audio" | "notifications" | "bluetooth" | "wifi" | "energy" | "stats"`
- `focusedScreenName: string`
- `barExpandedForDashboard: bool`
- `doNotDisturbEnabled: bool`
- `selectedAudioSinkId: string | null`
- `selectedAudioSourceId: string | null`
- `selectedNetworkId: string | null`
- `selectedBluetoothDeviceId: string | null`

Recommended state machine:

- `closed`
- `opening(tab)`
- `open(tab)`
- `switching(fromTab, toTab)`
- `closing`

Transition rules:

- Clicking a right-side indicator while closed opens the dashboard on the matching tab.
- Clicking the bluetooth indicator opens the `bluetooth` tab.
- Clicking the wifi or lan indicator opens the `wifi` tab.
- Clicking a different indicator while open switches tabs without closing the dashboard.
- Clicking the active indicator while open closes the dashboard.
- While the dashboard is open, the right-side indicators spread across the width of the sidebar and act as the visible tab strip.
- The active indicator receives a highlighted state.
- The highlighted bar state and the dashboard tab state must never drift apart.
- The left-to-right page order must match the top-bar indicator strip so tab changes slide horizontally in the expected direction.

## Top bar

- The bar is full width on each screen.
- The layout is left, center, right.
- The right module is compact while closed and expands to the dashboard width while the dashboard is active.

### Left module

- Show Hyprland workspaces for the current screen in numeric order.
- Format workspaces as `[workspace number] [window icon 1] [window icon 2] ...`.
- Show up to 2 window icons by default.
- If more windows exist, show `+n` after the visible icons.
- The focused workspace must be clearly highlighted.
- Occupied workspaces remain visible.
- Empty inactive workspaces may be hidden unless explicitly configured otherwise.
- Left-clicking a workspace focuses it.
- Left-clicking a window icon focuses that window.
- Middle-clicking a window icon closes that window.
- Right-click does nothing for now.

### Center module

- Show the current time.
- Default format is `HH:mm`.
- The clock stays visually centered independent of left and right content width.

### Right module

The right side contains status indicators for:

- overview or general shell status
- audio
- notifications
- bluetooth
- wifi or lan
- energy management
- system stats

Indicator rules:

- Indicators must be reusable components, not one-off tab buttons.
- Each indicator provides a glanceable summary state.
- Indicators double as tab selectors while the dashboard is visible.
- Use semantic Catppuccin colors for warning, critical, active, and muted states.

## Dashboard

- The dashboard is a right-aligned sidebar.
- It opens on the focused screen.
- It should visually connect to the right side of the top bar.
- Default width should stay in the `360-420px` range.
- The dashboard content is scrollable.
- The default generic open target is `overview`.

Tabs:

- overview
- audio
- notifications
- bluetooth
- wifi
- energy
- system stats

## Overview tab

The overview tab exposes the most common shell actions without going as deep as the dedicated tabs.

Required content:

- wifi enable or disable
- bluetooth enable or disable
- output volume slider
- input volume or input mute state
- brightness slider
- power mode selector
- session control for logout, shutdown, reboot, and hibernate
- battery summary when hardware exists
- active network summary
- active audio device summary
- notification count and Do Not Disturb state

Behavior rules:

- Prefer compact grouped controls over large detailed lists.
- Session-ending actions require confirmation.

## Audio tab

Required features:

- set default output sink
- set default input source
- list inputs
- list outputs or sinks
- show active streams per sink
- per-stream volume and mute controls
- drag and drop streams between sinks

Per input row:

- name
- volume slider
- mute button
- default marker if applicable

Per sink row:

- basic info
- default marker if applicable
- volume slider
- mute button
- list of streams currently routed to that sink

Per stream row:

- app icon
- app name or stream name
- volume slider
- mute button
- drag handle or equivalent affordance

Interaction rules:

- Use stable row ordering so dragging or expanding does not scramble the view.
- Valid drop targets highlight during drag.
- Cursor shape must communicate drag-move behavior.
- Use backend-supported volume range up to `150%` where available.

## Notifications tab

Required features:

- enable or disable notification toasts
- clear all
- newest-first notification list
- per-notification dismiss
- support notification actions and default action on click

Per notification:

- optional image support
- app icon
- title
- body
- markdown support for the body

Markdown scope:

- emphasis
- links
- inline code
- bullet lists
- line breaks

Rules:

- Urgent notifications should stand out.
- Empty state must be explicit.
- Long content should wrap cleanly and support collapsed or expanded display.

## Wifi tab

Required features:

- current connectivity status
- wifi enabled state
- active connection shown first
- disconnect action
- forget action for known networks
- known available networks section
- other available networks section
- inline password entry for protected unknown networks
- inline error rendering for connection failures

Per connection:

- SSID or display name
- signal strength
- security type
- connection state
- whether the network is known
- connect action
- disconnect action when active
- forget action when applicable

Advanced details may include:

- device name
- adapter state
- signal percentage
- security label
- autoconnect state
- associated profile names

## Bluetooth tab

Required features:

- adapter enabled state
- scan or discovery state
- connected devices first
- paired or known devices next
- newly discovered devices after that

Per device:

- icon
- display name
- device type
- connection state
- battery if available

Typical actions:

- enable or disable adapter
- start or stop scanning
- pair
- trust
- connect
- disconnect
- forget or remove

Advanced details may include:

- address
- adapter name
- paired, bonded, trusted, or blocked state
- battery status

## Energy tab

Required features:

- battery summary when present
- brightness slider
- power profile selector

Battery details:

- percentage
- charging, discharging, or full state
- estimated time to full or empty when available

Power profile rules:

- expose `power-saver`, `balanced`, and `performance` when supported
- hide or disable unsupported profiles cleanly

Brightness rules:

- if no brightness backend exists, show a clear unavailable state instead of a broken slider

## System stats tab

Act as a lightweight system monitor.

Required metrics:

- CPU usage
- memory usage
- swap usage
- disk usage for important mounts
- network throughput

Optional metrics when supported:

- GPU usage
- temperatures
- battery power draw
- charge or discharge rate
- per-core CPU detail

Rules:

- Prefer `1s` refresh for live metrics and slower intervals for expensive data.
- Keep graphs subtle and low-noise.
- Readability matters more than flashy telemetry.

## Reusable components

Prefer extending shared components over building each tab from scratch.

Build or reuse:

- `ShellState`
- `ActionButton`
- `StatusIcon`
- `SelectView`
- `DashboardTabButton`
- `DashboardPage`
- `DashboardPageHeader`
- `DashboardSection`
- `DashboardSectionHeader`
- `DashboardSectionContent`
- `InfoRow`
- `DeviceRow`
- `StatRow`
- `LabeledSlider`
- `ToggleTile`
- `ConfirmActionButton`
- `EmptyState`
- `Badge`
- `InlinePasswordField`
- `DragTargetHighlight`

Keep row sizes, icon sizes, text sizes, and internal gaps tokenized instead of repeating magic numbers per tab.
