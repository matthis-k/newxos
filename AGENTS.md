# AGENTS

Practical operating manual for coding agents in this NixOS flake.

## Project purpose

NixOS flake (`newxos`) managing two personal machines (desktop, laptop) and a live USB installer. Uses `flake-parts`, `import-tree`, and `flake-file` for modular NixOS/Home Manager configuration. Correctness: flake evaluates, `nix flake check` passes, configured systems build.

## Repository map

- `modules/` — source of truth for flake behavior (recursively imported by `import-tree`)
- `configs/` — hand-written program configs (Hyprland, Kitty, Neovim, OpenCode, Quickshell)
- `secrets/` — encrypted sops-nix secrets (age)
- `docs/` — compact agent knowledge base (architecture, ADRs, pitfalls, contracts, playbooks)
- `packages/` — custom Nix packages
- `patches/` — patch files
- `flake.nix` — **generated** by `flake-file`; edit `modules/`, never this file directly
- `flake.lock` — pinned inputs

## Build, test, and check commands

| Command | Purpose |
|---|---|
| `nix develop "path:$PWD"` | Enter default dev shell |
| `nix flake show "path:$PWD"` | Show all flake outputs |
| `nix flake check "path:$PWD"` | Run flake checks (evaluation + pre-commit hooks) |
| `nix run "path:$PWD#write-flake"` | Regenerate `flake.nix` |
| `nix run "path:$PWD#fmt"` | Format repo (treefmt) |
| `nix run "path:$PWD#repo-gate"` | Pre-commit gate: write-flake → statix → fmt → flake check → hooks |
| `nix run "path:$PWD#install-git-hooks"` | Install managed pre-commit hooks |
| `newxos memory reindex` | Rebuild Basic Memory index |
| `newxos memory reset` | Reset and rebuild Basic Memory |

Use `"path:$PWD"` (not `.`) for local flake references — `.` fails in untracked checkouts.

## Coding rules

- Search order: Basic Memory → `rg docs/` → source inspection.
- Do not read or print secrets. Prefer checking paths, wiring, permissions, or filenames.
- Edit `modules/`, never `flake.nix` (generated file).
- Use `"path:$PWD"` for local Nix commands during agent work.
- After changing flake-file declarations, run `nix run "path:$PWD#write-flake"`.
- After adding/changing a flake output, verify with `nix flake show "path:$PWD"`.
- Stage only task-related files; avoid broad `git add .` in a dirty worktree.
- Write commit subjects in Conventional Commits form: `<type>[optional scope][!]: <description>`.
- Load task-appropriate skills before QML, Qt/C++, UI design, profiling, docs, or code review work.
- Prefer reading upstream NixOS/Home Manager/darwin options from `nixos_nix` tool over stale notes.
- Keep exact option values, package lists, and generated settings in source files or nearby comments, not duplicated in docs.

## Architecture invariants

- `flake.nix` is generated by `flake-file`. All behavioral changes go in `modules/`.
- `import-tree` recursively imports the entire `modules/` tree for module discovery.
- `docs/` indexes concepts to source locations and records durable decisions. Source files are the truth for exact behavior.
- Basic Memory indexes `docs/` for semantic search.
- Feature directories in `modules/` follow the dendritic pattern (grouped by feature, not by type).
- Keep docs focused on indirection, placement rules, durable decisions, and library references. Do not mirror source values.

## Agent workflow

1. Search Basic Memory first for relevant knowledge before reading files or running commands.
2. Inspect before editing. Prefer minimal diffs — avoid broad rewrites unless requested.
3. Update docs and tests when behavior changes.
4. Run the narrowest relevant check first (`write-flake` → `fmt` → `flake check`).
5. Report skipped checks explicitly.
6. When a task reveals a repeatable mistake, append to `docs/pitfalls.md`.
7. When a task adds/changes a major tool, update the relevant docs and cross-link related pages.

## Safety / destructive actions

Require explicit user confirmation before:

- Deleting large files or directories
- Rotating secrets or reading secret payloads
- Rewriting git history
- Adding large dependencies
- Manually editing `flake.nix` (it is generated)
- Running destructive commands (`nix-collect-garbage`, `rm -rf`, `git push --force`, etc.)
- Broad `git add .` in a dirty worktree (stage only task files)
